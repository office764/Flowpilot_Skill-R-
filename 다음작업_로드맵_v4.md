### 작업 1: 모델 이미지 생성 — Flux Kontext Body 채우기

**현재 상태**: `AI 모델 이미지 생성 (Flux Kontext)` 노드의 Send Body가 비어있다.

**해야 할 작업**: Code in JavaScript1의 출력에서 `model_image_prompt`를 Flux Kontext API의 body에 전달해야 한다.

**구체적 방법**:
1. `AI 모델 이미지 생성 (Flux Kontext)` 노드를 연다
2. Send Body 섹션에서 Specify Body → **JSON** 선택
3. JSON Body 필드에 아래 값을 입력:

```json
={
  "prompt": "{{ $json.model_image_prompt }}",
  "model": "flux-kontext",
  "aspect_ratio": "9:16",
  "num_images": 1
}
```

**주의**: Flux Kontext API의 실제 파라미터명은 kie.ai 문서를 확인해야 한다. 위는 일반적인 구조이며, 실제 API 스펙에 따라 `prompt`, `model`, `input.prompt` 등으로 달라질 수 있다. 현재 v3에서 이 노드가 작동한 적이 있다면, 그때의 body 구조를 참고한다.

**연결**: Code in JavaScript1 → (첫 번째 item의 model_image_prompt만 필요) → 모델 이미지 체크 또는 직접 Flux Kontext

### 작업 2: TTS 경로 재설계 — Typecast 비동기 이슈 수정

**현재 상태**: `세그먼트별 TTS 생성 (Typecast)` 노드가 `responseFormat: "file"`로 설정되어 있다.

**문제**: Typecast v1/text-to-speech API는 **비동기(async)** API다. 요청 시 즉시 오디오 파일을 반환하지 않고, `speak_v2_id`를 담은 JSON을 반환한다. `responseFormat: "file"`로 설정하면 이 JSON 응답을 바이너리 파일로 오인하여 깨진 데이터가 나온다. 이것이 **TTS에서 음성이 안 나오고 배경음악만 나오는 근본 원인**이다.

**해야 할 작업** (새 노드 3개 추가, 기존 노드 1개 설정 변경):

**Step 2-1**: `세그먼트별 TTS 생성 (Typecast)` 노드의 Options 변경
- 노드를 열고 → Options → Response → Response Format 을 **"file"에서 "json"으로** 변경
- 이것만으로는 기존 워크플로우를 "건드리는" 것이 아니라 설정값 하나를 수정하는 것임

**Step 2-2**: TTS 결과 대기 노드 추가 (새 노드)
- `세그먼트별 TTS 생성 (Typecast)` 다음에 **Wait** 노드를 추가
- 노드명: `TTS 생성 대기 12초`
- Wait 시간: **12초**

**Step 2-3**: TTS 오디오 다운로드 URL 폴링 노드 추가 (새 노드)
- Wait 노드 다음에 **HTTP Request** 노드를 추가
- 노드명: `TTS 오디오 조회`
- Method: **GET**
- URL: `=https://api.typecast.ai/v1/text-to-speech/{{ $('세그먼트별 TTS 생성 (Typecast)').item.json.result.speak_v2_id }}`
- Authentication: 기존 Typecast 크리덴셜 사용 (httpHeaderAuth → Typecast TTS)

**Step 2-4**: TTS 오디오 URL 추출 Code 노드 수정 또는 새 노드 추가
- 폴링 결과에서 `audio_download_url`을 추출
- 이 URL을 기존 `세그먼트 오디오 업로드 (tmpfiles)` 또는 직접 InfiniteTalk에 전달

**연결 변경**:
```
현재: Typecast TTS → tmpfiles 업로드 → 오디오 URL 추출
변경: Typecast TTS → TTS 대기 12초 → TTS 오디오 조회 → (오디오 URL 추출 후) tmpfiles 업로드 → 오디오 URL 추출
```

### 작업 3: 스크립트 세그먼트 분할 → 디렉터 AI 출력 연결

**현재 상태**: `스크립트 세그먼트 분할` Code 노드가 `$('대본 파싱').json.script.lines[]`를 읽어서 segment 데이터를 만든다.

**문제**: 디렉터 AI의 Code in JavaScript1이 이미 5개 item을 section별로 분할하여 출력하고 있다. 따라서 `스크립트 세그먼트 분할` 노드가 **이중 분할**을 하거나, 아예 필요 없어질 수 있다.

**해야 할 작업** (2가지 옵션):

**옵션 A: Code in JavaScript1의 출력을 직접 TTS 루프에 연결** (권장)
- Code in JavaScript1의 output(5개 items)을 `TTS 세그먼트 순차 루프`에 직접 연결
- `스크립트 세그먼트 분할` 노드는 우회(bypass)
- TTS 루프에서 `$json.tts_text`를 Typecast에 전달

**옵션 B: 스크립트 세그먼트 분할 노드를 Bridge 노드로 교체**
- 기존 `스크립트 세그먼트 분할` 노드의 코드를 수정하여 디렉터 AI 출력 형식을 읽도록 변경
- 이 방법은 기존 노드를 "건드리는" 것이므로 권장하지 않음

**권장**: 옵션 A. Code in JavaScript1 → TTS 세그먼트 순차 루프 직접 연결.

### 작업 4: InfiniteTalk — 하드코딩 프롬프트를 동적 프롬프트로 교체

**현재 상태**: `토킹 아바타 생성 (InfiniteTalk)` 노드의 JSON Body에 프롬프트가 하드코딩되어 있다:
```
"prompt": "A natural-looking young Korean woman aged 24 casually talking in her bathroom..."
```

**해야 할 작업**: 이 하드코딩 프롬프트를 디렉터 AI가 생성한 `infinitetalk_prompt`로 교체한다.

**구체적 방법**:
1. `토킹 아바타 생성 (InfiniteTalk)` 노드를 연다
2. JSON Body의 `"prompt"` 값을 아래로 교체:

```
"prompt": "{{ $json.infinitetalk_prompt }}"
```

이렇게 하면 각 세그먼트의 토킹 클립이 해당 섹션에 맞는 고유한 장면 묘사로 생성된다.

**주의**: `$json.infinitetalk_prompt`가 이 노드까지 제대로 전달되려면, 앞서 `토킹 클립 데이터 정리` 노드가 이 필드를 포함하여 전달해야 한다. 현재 이 노드는 `image_url`, `audio_url`, `text` 정도만 전달하므로 `infinitetalk_prompt` 필드도 추가해야 한다.

**토킹 클립 데이터 정리 노드 수정 필요 부분**:
```javascript
// 기존 코드에서 segments 배열 안에 추가:
segments.push({
  segment_id: segId,
  section: item.json.section,
  text: item.json.text,
  direction: item.json.direction || '',
  audio_url: item.json.audio_url,
  image_url: imageUrl,
  infinitetalk_prompt: item.json.infinitetalk_prompt || ''  // ← 이 줄 추가
});
```

### 작업 5: B-Roll — 하드코딩 Code 노드를 동적 프롬프트로 교체

**현재 상태**: `B-Roll 프롬프트 5개 생성` Code 노드가 `$('대본 파싱').json.visual_scenes[]`를 읽어서 B-Roll 프롬프트를 직접 생성한다. visual_scenes가 없으면 fallback으로 하드코딩된 7개 씬을 사용한다.

**문제**: 디렉터 AI가 이미 각 섹션별 `broll_prompt`를 생성했는데, 이 Code 노드가 그걸 무시하고 자체적으로 다시 프롬프트를 만들고 있다.

**해야 할 작업**: B-Roll 프롬프트를 디렉터 AI의 `broll_prompt`에서 직접 가져오도록 변경한다.

**구체적 방법** (2가지 옵션):

**옵션 A: B-Roll 프롬프트 5개 생성 노드를 새 Bridge 노드로 교체** (권장)
- 기존 Code 노드 내용을 아래로 교체:

```javascript
const items = $input.all();
const scriptData = $('대본 파싱').first().json;

// 토킹 클립 결과 수집 (이전 루프에서 전달됨)
const talkingClips = items.map((item, i) => {
  let videoUrl = '';
  try {
    const rd = item.json.data || item.json;
    if (rd.response && rd.response.resultUris) {
      videoUrl = rd.response.resultUris[0];
    } else if (rd.resultJson) {
      const parsed = typeof rd.resultJson === 'string' ? JSON.parse(rd.resultJson) : rd.resultJson;
      videoUrl = parsed.video?.url || parsed.url || parsed.output?.[0] || '';
    }
  } catch(e) {
    videoUrl = 'parse_error: ' + e.message;
  }
  return { segment: i + 1, video_url: videoUrl };
});

// 디렉터 AI의 Code in JavaScript1 출력에서 broll 데이터 가져오기
const directorItems = $('Code in JavaScript1').all();

const brolls = directorItems.map((item, i) => ({
  json: {
    id: 'broll_' + (i + 1),
    name: item.json.broll_name || ('B-Roll ' + (i + 1)),
    prompt: item.json.broll_prompt,
    generation_type: 'TEXT_2_VIDEO',
    talking_clips: talkingClips,
    brand: scriptData.brand || item.json.brand,
    product: scriptData.product || item.json.product,
    full_script: scriptData.script?.full_script || item.json.full_script || ''
  }
}));

return brolls;
```

**옵션 B: B-Roll 순차 루프를 우회하고 Code in JavaScript1에서 직접 연결**
- 복잡도가 높아 비권장

### 작업 6: Typecast TTS의 emotion 파라미터 연결

**현재 상태**: Typecast JSON Body의 `"emotion_type": "smart"`가 하드코딩되어 있다.

**해야 할 작업**: 디렉터 AI가 생성한 `tts_emotion`(excited, empathetic, surprised, confident, warm)을 Typecast의 emotion 파라미터에 동적으로 전달한다.

**주의**: Typecast API가 이 emotion 값들을 직접 지원하는지 확인이 필요하다. Typecast의 `emotion_type` 파라미터가 수용하는 값이 `smart`, `happy`, `sad` 등 제한적일 수 있다. 이 경우 매핑 테이블이 필요:

| 디렉터 AI tts_emotion | Typecast emotion_type 매핑 |
|---------------------|--------------------------|
| excited | happy |
| empathetic | sad (또는 smart) |
| surprised | happy |
| confident | smart |
| warm | smart |
| curious | smart |
| urgent | smart |

Typecast API 문서에서 지원하는 emotion_type 목록을 확인한 후 매핑을 확정한다.

## PART 4: 작업 우선순위 및 실행 순서

| 순서 | 작업 | 난이도 | 영향도 | 소요 시간 |
|------|------|:------:|:------:|:---------:|
| **1번째** | 작업 2: TTS 비동기 수정 | 중 | 매우 높음 | 15분 |
| **2번째** | 작업 4: InfiniteTalk 동적 프롬프트 | 하 | 높음 | 5분 |
| **3번째** | 작업 5: B-Roll 동적 프롬프트 | 중 | 높음 | 10분 |
| **4번째** | 작업 1: Flux Kontext Body | 하 | 중 | 5분 |
| **5번째** | 작업 3: 세그먼트 분할 연결 | 중 | 높음 | 10분 |
| **6번째** | 작업 6: TTS emotion 매핑 | 하 | 낮음 | 5분 |

### 왜 이 순서인가?

1. **TTS 먼저**: TTS가 안 되면 InfiniteTalk에 audio_url을 전달할 수 없다. InfiniteTalk는 image + audio를 받아서 lip-sync 영상을 만드는데, audio가 깨져있으면 전체 파이프라인이 무너진다.

2. **InfiniteTalk 다음**: TTS가 정상 작동하면, 하드코딩 프롬프트를 동적 프롬프트로 교체하는 건 JSON Body의 문자열 하나만 바꾸면 된다.

3. **B-Roll 그 다음**: InfiniteTalk(토킹 영상)이 완성된 후에 B-Roll이 실행되므로, 토킹 영상이 정상 작동하는 것을 확인한 후 B-Roll을 수정하는 것이 안전하다.
